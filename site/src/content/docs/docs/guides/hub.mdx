## What is the Hub?

The Hub is xNet's server-side component. In a local-first system, the server is optional — it exists to improve availability, not to own your data. The Hub serves three purposes:

1. **Signaling** — Broker WebRTC handshakes so peers can find each other
2. **Relay** — Forward Yjs updates over WebSocket when direct P2P fails (firewalls, NAT)
3. **Persistence** (planned) — Store Y.Doc state so peers that were never online simultaneously can still sync

Today, xNet ships a **signaling server** that handles purposes 1 and 2. The full Hub with persistence is planned.

## Current architecture

```
Peer A ←→ Signaling Server ←→ Peer B
              (stateless relay)
```

The signaling server is entirely stateless. It never stores your data — it only forwards opaque messages between peers subscribed to the same room. All intelligence (sync protocol, signing, verification, rate limiting, peer scoring) lives in the clients.

## How signaling works

The signaling server implements the y-webrtc pub/sub protocol over WebSocket:

```
Client → Server:  { type: 'subscribe', topics: ['room1'] }
Client → Server:  { type: 'publish', topic: 'room1', data: { ... } }
Server → Others:  { type: 'publish', topic: 'room1', data: { ... } }
```

Each document gets a room named `xnet-doc-{nodeId}`. When a client publishes to a room, the message is broadcast to all other subscribers. The server doesn't inspect the message content.

### Endpoints

| Endpoint          | Purpose                                         |
| ----------------- | ----------------------------------------------- |
| `ws://host:4444/` | WebSocket connection                            |
| `GET /health`     | JSON health check (status, uptime, connections) |
| `GET /metrics`    | Prometheus-format metrics                       |

## Self-hosting the signaling server

### Run directly

```bash
cd infrastructure/signaling
pnpm install && pnpm build
PORT=4444 node dist/server.js
```

### Run with Docker

```bash
cd infrastructure/signaling
docker build -t xnet-signaling .
docker run -p 4444:4444 xnet-signaling
```

### Deploy to Fly.io

```bash
cd infrastructure/signaling
fly launch --name my-xnet-signaling
fly deploy
```

The included `fly.toml` configures auto-HTTPS, health checks, and Prometheus metrics.

### Configuration

| Variable  | Default | Description              |
| --------- | ------- | ------------------------ |
| `PORT`    | `4444`  | WebSocket and HTTP port  |
| `VERSION` | `0.0.1` | Reported in health check |

### Connect your app

Point the Electron app or web client to your signaling server:

```ts
<XNetProvider config={{
  signalingServers: ['wss://my-hub.example.com'],
  // ...
}}>
```

In the Electron app, the signaling URL is configurable in Settings.

## Hub roadmap

The Hub is designed to evolve through three operational modes:

| Mode                  | Capability                                               | Status  |
| --------------------- | -------------------------------------------------------- | ------- |
| **Signaling only**    | WebRTC handshake brokering. No data persistence.         | Shipped |
| **Signaling + Relay** | Also relays Yjs updates when P2P fails.                  | Shipped |
| **Full Hub**          | Persistence, backup, query, schema registry, federation. | Planned |

### Planned Full Hub features

- **Y.Doc persistence** — SQLite storage so offline peers can sync later
- **Encrypted backup** — Content-addressed blob storage
- **Query engine** — SQLite FTS5 for full-text search
- **UCAN authentication** — Verify tokens on WebSocket connect
- **Schema registry** — Publish and resolve schemas
- **Peer discovery** — DID resolution and peer registry
- **Hub federation** — Hub-to-hub queries across instances

### Planned deployment

```bash
# One-liner install (planned)
npx @xnet/hub --port 4444 --data ~/.xnet-hub

# Or with Docker
docker run -p 4444:4444 -v hub-data:/data ghcr.io/xnet-dev/xnet-hub
```

Three modes planned:

```bash
npx @xnet/hub --mode signaling   # Minimal — signaling only
npx @xnet/hub --mode relay       # Signaling + relay
npx @xnet/hub --mode full        # Everything
```

## Why not TURN?

xNet uses a WebSocket relay instead of TURN for NAT traversal:

- TURN is essentially the same as WebSocket relay but more complex to operate
- WebSocket relay integrates naturally with the signaling server
- No need for separate TURN infrastructure
- All traffic is signed with Ed25519 regardless of transport
