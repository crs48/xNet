useNode

You will learn

  • All options and the full return type (grouped by category)
  • How to bind a Yjs document to TipTap
  • How the dual-write model works (NodeStore + Y.Doc)
  • Sync lifecycle, presence, and error handling

What useNode replaces

In a traditional app, useNode replaces:

  • WebSocket connection — real-time sync per document
  • Yjs provider setup — WebRTC/WebSocket provider, awareness, persistence
  • Autosave logic — debounced persistence
  • Presence system — who’s online, cursor positions
  • Document loading — fetch, cache, conflict handling

One hook. All of it.

The dual-write model

Section titled “The dual-write model”

This is the most important thing to understand about useNode:

When you call update({ title: 'New Title' }), it writes to two places:

  1. NodeStore — structured data with Lamport timestamps (for useQuery and persistence)
  2. Y.Doc meta map — a Y.Map('meta') inside the Yjs document (for P2P sync)

When a remote peer changes the Y.Doc meta map, a metaObserver picks up the change and writes it back to the NodeStore. This bidirectional bridge keeps both stores in sync.

Direct Y.Doc edits don’t update NodeStore

If you modify the Y.Doc’s text content directly (e.g., via TipTap), that content syncs fine through Yjs. But if you modify the Y.Doc’s meta map directly (bypassing update()), the change will NOT be written to the NodeStore unless it comes from a remote peer. Always use the update() method for structured properties.

Lifecycle

Section titled “Lifecycle”
  1. Mount — NodeStore lookup. If not found and createIfMissing is set, create the node.
  2. Init Y.Doc — If schema has document: 'yjs', load stored content and initialize the Yjs document.
  3. Connect — Acquire a sync connection from SyncManager (or create a WebSocketSyncProvider as fallback).
  4. Editing — User and peer changes flow through Yjs CRDT. Property changes go through the dual-write bridge.
  5. Persist — Y.Doc changes are debounced and saved to SQLite as Y.encodeStateAsUpdate().
  6. Unmount — Flush all pending changes immediately, release the sync connection.

Navigation flush

When the component unmounts, useNode immediately flushes the Y.Doc to storage. If you navigate back to the same node quickly, the new instance waits for the previous flush to complete before loading. This prevents data loss but can cause a brief delay.
